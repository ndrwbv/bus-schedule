# Фича: Авторизация (Email + VK OAuth)

**Ветка:** `11-authentication`
**Создана:** 2026-03-29
**Статус:** черновик

## Контекст

**Зачем:** Сейчас пользователи анонимны (UUID в localStorage). Для развития проекта нужна авторизация:
- Жалобы с текстом — только от авторизованных (борьба со спамом, ответственность за написанное)
- Подписки на остановки/маршруты (push-уведомления, будущая фича)
- Платные фичи (live-отслеживание автобуса в будущем) — нужен аккаунт для привязки оплаты
- Персонализация — избранные остановки привязаны к аккаунту, а не к браузеру

**Что есть сейчас:**
- Анонимный UUID в localStorage (используется для rate limit жалоб)
- Нет понятия «пользователь» на бэкенде
- Жалобы отправляются анонимно с предсозданными шаблонами
- Избранные остановки хранятся только в localStorage

## User Stories

### US-1 — Авторизация через Email (magic link) (Приоритет: P1)

Пользователь вводит email, получает ссылку для входа на почту, кликает — авторизован. Без паролей, максимально просто.

**Почему этот приоритет:** Email — универсальный способ авторизации, есть у всех. Magic link проще для пользователя чем пароль (целевая аудитория — обычные люди, пользователи автобуса).

**Как проверить независимо:** Ввести email → получить письмо → кликнуть ссылку → на сайте авторизован, видно имя/email в хедере.

**Сценарии приёмки:**

1. **Дано** пользователь не авторизован, **Когда** нажимает «Войти» и вводит email, **Тогда** на email приходит ссылка для входа (magic link), на сайте показывается «Проверьте почту»
2. **Дано** пользователь получил magic link, **Когда** кликает по ссылке, **Тогда** перенаправляется на сайт и авторизован (сессия создана)
3. **Дано** magic link старше 15 минут, **Когда** пользователь кликает, **Тогда** видит «Ссылка устарела, запросите новую»
4. **Дано** пользователь авторизован, **Когда** обновляет страницу, **Тогда** остаётся авторизованным (токен в cookie/localStorage)
5. **Дано** пользователь авторизован, **Когда** нажимает «Выйти», **Тогда** сессия удалена, пользователь анонимен

---

### US-2 — Авторизация через VK OAuth (Приоритет: P1)

Пользователь нажимает «Войти через ВКонтакте», перенаправляется на VK, подтверждает доступ, возвращается авторизованным. Основная аудитория — Томск, VK популярнее других соцсетей.

**Почему этот приоритет:** VK — основная соцсеть целевой аудитории (Томск, Россия). Многие пользователи предпочтут войти через VK чем вводить email.

**Как проверить независимо:** Нажать «Войти через VK» → авторизоваться в VK → вернуться на сайт авторизованным.

**Сценарии приёмки:**

1. **Дано** пользователь не авторизован, **Когда** нажимает «Войти через VK», **Тогда** перенаправляется на страницу авторизации VK OAuth
2. **Дано** пользователь подтвердил доступ в VK, **Когда** VK перенаправляет обратно на сайт, **Тогда** пользователь авторизован, имя и аватар подтянуты из VK
3. **Дано** пользователь ранее входил через VK, **Когда** нажимает «Войти через VK» снова, **Тогда** авторизация происходит без повторного подтверждения (если сессия VK активна)
4. **Дано** пользователь отказал в доступе на странице VK, **Когда** VK перенаправляет обратно, **Тогда** пользователь видит «Авторизация отменена», остаётся неавторизованным

---

### US-3 — Привязка нескольких способов входа к одному аккаунту (Приоритет: P2)

Если пользователь вошёл через email, а потом через VK (или наоборот) — аккаунты объединяются, если email совпадает.

**Почему этот приоритет:** Улучшает UX, но не блокирует базовую авторизацию. Без этого — два отдельных аккаунта.

**Как проверить независимо:** Войти через email X → выйти → войти через VK (где тот же email X) → данные (жалобы, избранное) сохранены.

**Сценарии приёмки:**

1. **Дано** пользователь зарегистрирован через email, **Когда** входит через VK с тем же email, **Тогда** VK привязывается к существующему аккаунту
2. **Дано** пользователь зарегистрирован через VK, **Когда** входит через email (тот же что в VK), **Тогда** email привязывается к существующему аккаунту
3. **Дано** email в VK не совпадает ни с одним аккаунтом, **Тогда** создаётся новый аккаунт

---

### US-4 — Свободный текст в жалобах для авторизованных (Приоритет: P1)

Авторизованные пользователи могут писать свободный текст в жалобах (textarea). Анонимные — только шаблоны.

**Почему этот приоритет:** Это основная причина добавления авторизации на первом этапе. Текстовые жалобы от анонимов — путь к спаму.

**Как проверить независимо:** Войти → открыть жалобы → появляется тогл «Написать подробнее» → отправить текст. Без авторизации — тогла нет.

**Сценарии приёмки:**

1. **Дано** пользователь авторизован и на странице жалоб, **Когда** смотрит на кнопки жалоб, **Тогда** видит тогл «Написать подробнее» под шаблонами
2. **Дано** пользователь не авторизован, **Когда** смотрит на кнопки жалоб, **Тогда** тогл «Написать подробнее» не отображается, вместо него — подсказка «Войдите чтобы написать подробнее»
3. **Дано** пользователь авторизован и написал текст жалобы, **Когда** отправляет, **Тогда** жалоба сохраняется с `message` и `user_id` (привязана к аккаунту)

---

### US-5 — UI авторизации (Приоритет: P1)

Кнопка «Войти» в хедере. Модальное окно / bottom sheet с выбором способа входа. После авторизации — аватар/имя + кнопка «Выйти».

**Почему этот приоритет:** Без UI пользователи не смогут авторизоваться.

**Как проверить независимо:** Открыть сайт → нажать «Войти» → увидеть выбор Email/VK → авторизоваться → увидеть своё имя в хедере.

**Сценарии приёмки:**

1. **Дано** пользователь не авторизован, **Когда** нажимает «Войти» в хедере, **Тогда** открывается bottom sheet с кнопками «Войти через VK» и «Войти через Email»
2. **Дано** пользователь авторизован, **Когда** смотрит на хедер, **Тогда** видит своё имя (или email) и кнопку «Выйти»
3. **Дано** мобильное устройство, **Когда** пользователь нажимает «Войти», **Тогда** bottom sheet адаптирован под мобильный экран

## Ключевые сущности

- **User**: `id` (UUID), `email` (unique, nullable), `name` (nullable), `avatar_url` (nullable), `vk_id` (unique, nullable), `created_at`, `updated_at`
- **AuthToken**: `id`, `user_id` (FK → User), `token` (unique), `expires_at`, `created_at` — JWT refresh tokens / session tokens
- **MagicLink**: `id`, `email`, `token` (unique), `expires_at`, `used` (boolean), `created_at` — одноразовые ссылки для входа через email

## Edge Cases

- Что если пользователь не получил magic link? → Кнопка «Отправить ещё раз» (rate limit: 1 письмо в 60 секунд)
- Что если VK API недоступен? → Показать ошибку «ВКонтакте недоступен, попробуйте позже или войдите через Email»
- Что если пользователь удалил аккаунт VK? → Аккаунт на severbus.ru остаётся, можно войти через email (если привязан)
- Что если email из VK не подтверждён? → Не использовать для автоматической привязки аккаунтов, создать отдельный аккаунт
- Что если токен авторизации украден? → JWT с коротким TTL (15 мин access + 7 дней refresh), refresh token одноразовый (ротация)
- Что если анонимный пользователь авторизуется — что с его данными? → Мигрировать localStorage UUID → связать жалобы с новым аккаунтом (best effort, не блокер)
- SMTP-сервис упал? → Показать «Не удалось отправить письмо, попробуйте VK или позже». Логировать ошибку.

## Безопасность

- Magic link: одноразовый, TTL 15 минут, rate limit на отправку
- JWT: access token (15 мин, в памяти) + refresh token (7 дней, httpOnly cookie)
- VK OAuth: проверка `state` параметра (CSRF protection)
- CORS: только severbus.ru
- Rate limit: авторизация — 5 попыток в 15 минут на IP
- Хранение: пароли не хранятся (magic link), VK токены не хранятся (только vk_id)

## Зависимости от других спек

- **Спека 01 (бэкенд + деплой)** — нужен работающий бэкенд
- **Спека 03 (жалобы)** — US-4 расширяет жалобы текстом для авторизованных
- **Спека 04 (live-отслеживание)** — в будущем платная фича, привязка к аккаунту
- **Спека 08 (донаты)** — привязка оплаты к аккаунту

## Инфраструктура

- **SMTP**: Yandex 360 / Brevo (бесплатный тир для малого объёма) для отправки magic link
- **VK OAuth**: Регистрация приложения на vk.com/dev, получение `client_id` и `client_secret`
- **Секреты**: `JWT_SECRET`, `VK_CLIENT_ID`, `VK_CLIENT_SECRET`, `SMTP_HOST`, `SMTP_USER`, `SMTP_PASS`

## Критерии приёмки

- [ ] Таблицы `users`, `magic_links` в SQLite
- [ ] POST `/api/auth/email` — отправка magic link на email
- [ ] GET `/api/auth/email/verify?token=...` — верификация magic link, создание сессии
- [ ] GET `/api/auth/vk` — редирект на VK OAuth
- [ ] GET `/api/auth/vk/callback` — обработка ответа VK, создание сессии
- [ ] POST `/api/auth/refresh` — обновление access token по refresh token
- [ ] POST `/api/auth/logout` — удаление сессии
- [ ] GET `/api/auth/me` — текущий пользователь (или 401)
- [ ] UI: кнопка «Войти» в хедере → bottom sheet с выбором Email/VK
- [ ] UI: после авторизации — имя/аватар в хедере + «Выйти»
- [ ] UI: тогл «Написать подробнее» в жалобах — только для авторизованных
- [ ] JWT access (15 мин) + refresh (7 дней) token ротация
- [ ] Rate limit на авторизацию (5 попыток / 15 мин на IP)
- [ ] [P2] Привязка нескольких способов входа к одному аккаунту (по совпадению email)
- [ ] [P2] Миграция анонимных данных (localStorage UUID → аккаунт) при первой авторизации
