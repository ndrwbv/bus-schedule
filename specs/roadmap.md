# Roadmap — severbus.ru

## Обзор

Возрождение severbus.ru: от статического SPA с захардкоженным расписанием к автоматизированному сервису с бэкендом, live-отслеживанием и монетизацией.

## Фазы

### Фаза 1: Фундамент (бэкенд + деплой)

> Цель: сайт работает на своём сервере, есть бэкенд для API

| # | Спека | Приоритет | Оценка |
|---|-------|-----------|--------|
| 01 | [Бэкенд + деплой](./01-backend-and-deploy.md) | P1 | 2-3 дня |

**Результат:** Express-сервер на Docker Compose, nginx с SPA + API proxy, SSL, CI/CD.

**Блокирует:** все последующие фичи (парсинг, жалобы, live, мониторинг).

---

### Фаза 2: Ядро (расписание + жалобы)

> Цель: расписание обновляется автоматически, жалобы работают

| # | Спека | Приоритет | Оценка |
|---|-------|-----------|--------|
| 02 | [Автоматический парсинг расписания](./02-schedule-parser.md) | P1 | 3-5 дней |
| 05 | [Переработка направлений](./05-directions-rework.md) | P1 (анализ) / P2 (рефакторинг) | 1-2 дня |
| 03 | [Жалобы](./03-complains.md) | P1 | 1-2 дня |

**Порядок:** 02 → 05 (зависит от результатов парсинга) → 03 (параллельно с 05).

**Результат:** Сайт показывает актуальное расписание из API, жалобы работают на своём бэкенде.

---

### Фаза 3: Live + мониторинг

> Цель: пользователи видят где автобус, автор видит что сервис жив

| # | Спека | Приоритет | Оценка |
|---|-------|-----------|--------|
| 04 | [Live-отслеживание на карте](./04-live-tracking.md) | P1 | 2-3 дня |
| 06 | [Мониторинг](./06-monitoring.md) | P1 | 2-3 дня |

**Порядок:** 04 и 06 можно делать параллельно.

**Результат:** Маркер автобуса на карте, дашборд мониторинга со статистикой.

---

### Фаза 4: Авторизация + персонализация

> Цель: пользователи могут авторизоваться, писать текстовые жалобы, подготовка к платным фичам

| # | Спека | Приоритет | Оценка |
|---|-------|-----------|--------|
| 11 | [Авторизация (Email + VK)](./11-authentication.md) | P1 | 3-5 дней |

**Порядок:** После 03 (жалобы) — авторизация расширяет жалобы текстом.

**Результат:** Авторизация через Email (magic link) и VK OAuth, текстовые жалобы для авторизованных, подготовка к подпискам и платным фичам.

---

### Фаза 5: Комьюнити + монетизация

> Цель: свои каналы связи с аудиторией, проект приносит деньги

| # | Спека | Приоритет | Оценка |
|---|-------|-----------|--------|
| 10 | [Каналы коммуникации (TG, VK, MAX)](./10-community-channels.md) | P1 | 1 день |
| 08 | [Донаты](./08-donations.md) | P1 | 0.5-1 день |
| 07 | [Реклама (РСЯ + локальная)](./07-ads.md) | P1 (РСЯ) / P2 (локальная) | 1-2 дня |
| 09 | [Привлечение локальных рекламодателей](./09-local-business-outreach.md) | P2 | 2-3 дня (бизнес) |

**Порядок:** 10 (каналы — можно сразу в Фазе 2, параллельно с разработкой) → 08 (донаты) → 07 (РСЯ после запуска) → 09 (когда есть статистика).

**Результат:** Свои каналы в TG/VK/MAX с фидбеком, кнопка донатов, РСЯ-баннер, первые обращения к локальным бизнесам.

## Timeline (оптимистичный)

```
Неделя 1:  [01] Бэкенд + деплой + [10] создать каналы TG/VK
Неделя 2:  [02] Парсер расписания + [05] анализ направлений
Неделя 3:  [03] Жалобы + [08] Донаты
Неделя 4:  [04] Live-отслеживание
Неделя 5:  [06] Мониторинг + [07] РСЯ
Неделя 6+: [09] Локальные рекламодатели
```

Каналы используются на протяжении всей разработки: каждый релиз → пост в канал → фидбек в комментариях.

## Зависимости

```
01 Бэкенд + деплой
├── 02 Парсер расписания
│   └── 05 Направления (анализ)
├── 03 Жалобы
├── 04 Live-отслеживание
├── 06 Мониторинг
└── 07 Реклама (бэкенд для локальных баннеров)

├── 11 Авторизация (Email + VK)
│   └── зависит от 01 (бэкенд) и 03 (жалобы)
│   └── блокирует платные фичи (04 live)

Независимые:
├── 10 Каналы TG/VK/MAX (можно сразу, фидбек на всём протяжении)
├── 08 Донаты (только фронт, можно раньше)
└── 09 Локальные рекламодатели (бизнес-задача, после запуска)
```

## Принципы

1. **Не повторять ошибку** — всё что можно автоматизировать должно быть автоматизировано
2. **Graceful degradation** — если какой-то API упал, сайт продолжает работать
3. **Минимальная стоимость поддержки** — SQLite вместо Postgres, бесплатные тиры, один сервер
4. **Фронт менять минимально** — формат `ISchedule` сохраняется, данные приходят из API вместо константы
