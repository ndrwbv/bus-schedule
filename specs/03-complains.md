# Фича: Жалобы (замена popooga.ru)

**Ветка:** `03-complains`
**Создана:** 2026-03-25
**Статус:** не начат

## Контекст

**Зачем:** Фронтенд для жалоб уже есть и работал хорошо (пользователи отмечали задержки и пропуски). Но бэкенд на popooga.ru мёртв — нужно поднять свой. Фронт-код минимально менять — только URL API.

**Что есть сейчас:**
- `src/features/Complains/` — полный UI: список жалоб, создание, шаблоны быстрых ответов
- `useComplains.tsx` — GraphQL запросы к `popooga.ru` (findComplains, createComplain)
- Типы жалоб: `earlier`, `later`, `not_arrive`, `passed_by`
- Поллинг каждые 5 секунд

## User Stories

### US-1 — REST API для жалоб (Приоритет: P1)

Бэкенд хранит жалобы в SQLite и отдаёт по REST API (вместо мёртвого GraphQL на popooga.ru).

**Почему этот приоритет:** Жалобы — ключевая фича, которую пользователи вспоминают первой.

**Как проверить независимо:** POST жалобу → GET список → жалоба в списке.

**Сценарии приёмки:**

1. **Дано** бэкенд запущен, **Когда** POST `/api/complains` с `{ stop, direction, type }`, **Тогда** жалоба сохранена, ответ 201
2. **Дано** есть жалобы за сегодня, **Когда** GET `/api/complains`, **Тогда** список жалоб за текущий день
3. **Дано** жалобы за вчера, **Когда** GET `/api/complains`, **Тогда** вчерашние не показываются (по умолчанию)

---

### US-2 — Фронтенд: переключение на REST API (Приоритет: P1)

Заменить GraphQL-вызовы к popooga.ru на fetch к `/api/complains`.

**Почему этот приоритет:** Без этого жалобы не работают на фронте.

**Сценарии приёмки:**

1. **Дано** пользователь на сайте, **Когда** нажимает «автобус не пришёл», **Тогда** жалоба уходит на свой бэкенд
2. **Дано** есть жалобы от других пользователей, **Когда** пользователь открывает сайт, **Тогда** видит актуальные жалобы

---

### US-3 — «Автобус пришёл» — положительный отзыв (Приоритет: P2)

Добавить возможность отметить что автобус пришёл вовремя. Это данные для краудсорсинга — показать остальным что автобус реально ходит.

**Почему этот приоритет:** Расширяет функционал, но не блокирует базовые жалобы.

**Сценарии приёмки:**

1. **Дано** пользователь на сайте, **Когда** нажимает «автобус пришёл», **Тогда** отметка сохранена
2. **Дано** автобус пришёл по отметке другого пользователя, **Когда** другой пользователь смотрит расписание, **Тогда** видит «автобус пришёл 5 мин назад (отметили пользователи)»

---

### US-4 — Агрегированная статистика жалоб (Приоритет: P1)

Рядом с кнопками жалоб показывать сводку за сегодня: «12 человек отметили что автобус не пришёл на Интернационалистов». Даёт ощущение что ты не один — ради этого люди и пишут в группы.

**Почему этот приоритет:** Это ключевая мотивация отправлять жалобы — видеть что другие тоже жалуются. Без этого кнопка ощущается как «отправить в пустоту».

**Сценарии приёмки:**

1. **Дано** 5 жалоб типа `not_arrive` на остановку «Интернационалистов» за сегодня, **Когда** пользователь выбрал эту остановку, **Тогда** видит «Сегодня 5 человек отметили что автобус не пришёл»
2. **Дано** 0 жалоб за сегодня, **Тогда** блок статистики не показывается
3. **Дано** жалобы на нескольких остановках, **Когда** пользователь переключает остановку, **Тогда** статистика обновляется для выбранной остановки

---

### US-5 — Свободный текст жалобы (Приоритет: P2)

Под кнопками быстрых жалоб — раскрываемое поле для свободного текста (*). По умолчанию скрыто, открывается по тоглу «Написать подробнее». Базовый фильтр мата.

**Почему этот приоритет:** Не все ситуации покрываются шаблонами. Иногда хочется написать «водитель проехал мимо остановки хотя я махал рукой». Но это P2 — шаблоны покрывают 90% случаев.

**Сценарии приёмки:**

1. **Дано** пользователь на сайте, **Когда** нажимает «Написать подробнее», **Тогда** раскрывается textarea для свободного текста
2. **Дано** textarea открыт, **Когда** пользователь вводит текст и отправляет, **Тогда** жалоба сохраняется с полем `message`
3. **Дано** пользователь ввёл текст с матом, **Когда** отправляет, **Тогда** мат заменяется звёздочками (базовый фильтр по словарю), жалоба сохраняется
4. **Дано** textarea открыт, **Когда** пользователь нажимает тогл снова, **Тогда** textarea скрывается
5. **Дано** другой пользователь смотрит жалобы, **Тогда** видит текст жалобы (с замаскированным матом) под стандартной меткой типа жалобы

---

## Ключевые сущности

- **Complain**: `id`, `stop` (название остановки), `direction` (inSP/out/inLB), `type` (earlier/later/not_arrive/passed_by/arrived), `message` (свободный текст, опционально), `user_id` (анонимный UUID), `created_at`
- **ComplainStats**: агрегация жалоб за сегодня по остановке и типу (для блока статистики)

## Edge Cases

- Спам жалобами? → Rate limit: 1 жалоба в минуту на user_id
- Что если user_id нет (новый пользователь)? → Генерировать UUID на фронте, хранить в localStorage
- Поллинг каждые 5 сек — не нагрузит ли бэкенд? → При 2К пользователях нормально, но можно увеличить интервал до 15-30 сек или перейти на SSE
- Мат в свободном тексте? → Базовый словарь матных слов, замена на `***`. Не блокировать отправку — просто маскировать. Словарь хранить на бэкенде.
- Очень длинные тексты? → Лимит 500 символов на textarea
- Пустой текст при открытом тогле? → Отправляется как обычная жалоба без текста
