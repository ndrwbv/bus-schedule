# Фича: Мониторинг

**Ветка:** `06-monitoring`
**Создана:** 2026-03-25
**Статус:** не начат

## Контекст

**Зачем:** Нужно видеть что сервис жив, сколько пользователей, работает ли API перевозчика, актуально ли расписание. Без мониторинга проблемы обнаруживаются только когда пользователи жалуются в группе.

**Что есть сейчас:** Только Google Analytics и Яндекс Метрика на фронте. Никакого серверного мониторинга.

## User Stories

### US-1 — Дашборд мониторинга (Приоритет: P1)

Страница `severbus.ru/health` (защищена паролем) показывает: статус сервера, загрузку CPU/памяти, статус БД, статус API перевозчика, дату последнего обновления расписания.

**Почему этот приоритет:** Без мониторинга не понятно, работает ли вообще всё.

**Как проверить независимо:** Открыть `severbus.ru/health`, ввести пароль, увидеть дашборд.

**Сценарии приёмки:**

1. **Дано** пользователь знает пароль, **Когда** открывает `/health`, **Тогда** видит дашборд с метриками
2. **Дано** пользователь не знает пароль, **Когда** открывает `/health`, **Тогда** форма ввода пароля
3. **Дано** API перевозчика недоступен, **Когда** смотрю дашборд, **Тогда** статус API — красный

---

### US-2 — Трекинг уникальных пользователей (Приоритет: P1)

Каждому пользователю при первом визите выдаётся анонимный UUID (в localStorage). Бэкенд считает уникальных пользователей за день/неделю/месяц.

**Почему этот приоритет:** Нужно знать сколько реально людей пользуется, чтобы показывать рекламодателям и отслеживать рост.

**Сценарии приёмки:**

1. **Дано** новый пользователь, **Когда** открывает сайт, **Тогда** генерируется UUID, сохраняется в localStorage, отправляется ping на бэкенд
2. **Дано** вернувшийся пользователь, **Когда** открывает сайт, **Тогда** используется существующий UUID, ping на бэкенд (не создаёт дубликат)
3. **Дано** дашборд мониторинга, **Когда** смотрю статистику, **Тогда** вижу: уников сегодня, за неделю, за месяц

---

### US-3 — Статус парсинга расписания (Приоритет: P1)

В дашборде: когда последний раз обновлено расписание, результат последнего парсинга (успех/ошибка), хеш текущего файла.

**Сценарии приёмки:**

1. **Дано** расписание обновлено 2 часа назад, **Когда** смотрю дашборд, **Тогда** вижу «Последнее обновление: 2 часа назад, статус: ОК»
2. **Дано** последний парсинг упал, **Тогда** вижу «Статус: ОШИБКА, причина: [описание]»

## Ключевые сущности

- **UserVisit**: `user_id` (UUID), `visited_at` (date), `page` (опционально)
- **HealthMetrics**: CPU, memory, disk, uptime, db_size, schedule_last_updated, live_api_status

## Edge Cases

- Что если пользователь очистил localStorage? → Новый UUID, считается как новый пользователь. Это ок для общей картины.
- Нагрузка от пингов? → Один ping при загрузке страницы, не при каждом действии
- Пароль к /health хранить где? → В .env на сервере, simple basic auth

## Критерии приёмки

- [ ] `/health` показывает дашборд с метриками (за паролем)
- [ ] CPU, память, uptime, размер БД
- [ ] Статус API перевозчика (live + расписание)
- [ ] Дата последнего обновления расписания
- [ ] Статистика пользователей: уники за день/неделю/месяц
- [ ] Фронт отправляет ping с user_id при загрузке
