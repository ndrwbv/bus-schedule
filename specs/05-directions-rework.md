# Фича: Переработка направлений

**Ветка:** `05-directions-rework`
**Создана:** 2026-03-25
**Статус:** черновик

## Контекст

**Зачем:** Сейчас направления захардкожены как `inSP` (в Северный парк), `out` (в город), `inLB` (в Левобережный). При переходе на автоматический парсинг расписания с пассажир-онлайн названия направлений и остановок могут не совпадать. Также пользователи из группы могут подсказать, какие направления реально актуальны. Нужно продумать как это лучше отображать.

**Что есть сейчас:**
- Enum `DirectionsNew`: `inSP`, `inLB`, `out`
- i18n: «В северный парк», «В город», «В левобережный»
- Три кнопки в `DirectionChanger`
- Списки остановок жёстко привязаны к направлениям: `StopsInSpOptions`, `StopsInLbOptions`, `StopsOutOptions`
- В жалобах (`getDirectionString`) нет ветки для `inLB` — потенциальный баг

## User Stories

### US-1 — Анализ направлений из нового расписания (Приоритет: P1)

Сравнить направления и остановки из нового Word-файла с текущими в коде. Определить: совпадают ли названия, есть ли новые направления, изменился ли набор остановок.

**Почему этот приоритет:** Без этого парсер может выдать данные, которые фронт не поймёт.

**Сценарии приёмки:**

1. **Дано** распаршенный Word-файл, **Когда** сравниваем с текущим `SCHEDULE`, **Тогда** документирован маппинг: какие остановки совпали, какие новые, какие исчезли
2. **Дано** маппинг готов, **Когда** появилась остановка с другим названием (например «пл. Ленина» vs «Площадь Ленина»), **Тогда** настроен алиас для совместимости

---

### US-2 — Динамические направления из API (Приоритет: P2)

Направления и остановки приходят с бэкенда вместе с расписанием. Фронтенд рендерит кнопки динамически, а не из захардкоженного enum.

**Почему этот приоритет:** Позволяет масштабироваться на новые маршруты без изменения фронта. Но на первом этапе можно обойтись маппингом.

**Сценарии приёмки:**

1. **Дано** API отдаёт расписание с 3 направлениями, **Когда** фронт рендерит кнопки, **Тогда** показывает 3 кнопки с названиями из API
2. **Дано** API отдаёт 2 направления (одно убрали), **Когда** фронт рендерит, **Тогда** показывает 2 кнопки

---

## Edge Cases

- Что если в новом расписании направления называются по-другому? → Маппинг на бэкенде при парсинге
- Что если добавилось 4-е направление? → UI должен масштабироваться (горизонтальный скролл или 2x2 grid)
- Что если остановка переименована? → Алиасы, чтобы избранные остановки пользователей не сломались
