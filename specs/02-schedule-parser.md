# Фича: Автоматический парсинг расписания

**Ветка:** `02-schedule-parser`
**Создана:** 2026-03-25
**Статус:** не начат

## Контекст

**Зачем:** Главная причина, по которой сайт был заброшен — расписание обновлялось руками. Перевозчик публикует расписание на сайте «пассажир-онлайн» в виде Word-файла. Нужно автоматически находить этот файл, скачивать, парсить и превращать в формат `ISchedule`, который уже понимает фронтенд.

**Что есть сейчас:** Расписание захардкожено в `src/shared/common/schedule.ts` как TypeScript-константа `SCHEDULE`. Формат:
```
direction → dayOfWeek → stopName → ["8:15", "10:25", ...]
```
Направления: `inSP`, `out`, `inLB`. Дни: `'0'`–`'6'` (Date.getDay()). Остановки — строки на русском.

## User Stories

### US-1 — Парсинг Word-файла с расписанием (Приоритет: P1)

Бэкенд скачивает Word-файл с сайта перевозчика, извлекает таблицы и превращает их в формат `ISchedule`.

**Почему этот приоритет:** Без этого расписание по-прежнему обновляется руками.

**Как проверить независимо:** Скормить парсеру реальный Word-файл → получить JSON в формате `ISchedule` → сравнить с ручным расписанием.

**Сценарии приёмки:**

1. **Дано** Word-файл с расписанием от перевозчика, **Когда** парсер обрабатывает файл, **Тогда** результат — JSON в формате `ISchedule` с корректными направлениями, днями и временами
2. **Дано** Word-файл содержит таблицу с нестандартным форматированием (объединённые ячейки, пустые строки), **Когда** парсер обрабатывает файл, **Тогда** парсер корректно извлекает данные или возвращает ошибку с описанием проблемы
3. **Дано** парсер не может распарсить файл стандартным способом, **Когда** включён AI-fallback, **Тогда** файл отправляется в GPT API для парсинга

---

### US-2 — Автоматическое обновление по расписанию (Приоритет: P1)

Cron-задача проверяет сайт перевозчика раз в день. Если файл изменился (по хешу или дате) — скачивает и парсит заново.

**Почему этот приоритет:** Без автоматического обновления снова придётся обновлять руками.

**Как проверить независимо:** Подождать сутки → проверить что в логах была попытка обновления → если файл не менялся, расписание не перезаписано.

**Сценарии приёмки:**

1. **Дано** cron настроен на ежедневный запуск, **Когда** наступает время, **Тогда** бэкенд проверяет файл на сайте перевозчика
2. **Дано** файл на сайте изменился (другой хеш), **Когда** cron сработал, **Тогда** новый файл скачан, распаршен, расписание в БД обновлено
3. **Дано** файл не изменился, **Когда** cron сработал, **Тогда** расписание не перезаписывается, в логе — «расписание актуально»

---

### US-3 — Ручной триггер обновления (Приоритет: P1)

Эндпоинт POST `/api/schedule/refresh` запускает парсинг вручную. Защищён паролем/токеном.

**Почему этот приоритет:** Если перевозчик внезапно обновил расписание — не ждать сутки.

**Сценарии приёмки:**

1. **Дано** авторизованный запрос, **Когда** POST `/api/schedule/refresh`, **Тогда** парсинг запускается немедленно, ответ — результат (успех/ошибка)
2. **Дано** неавторизованный запрос, **Когда** POST `/api/schedule/refresh`, **Тогда** ответ 401

---

### US-4 — API для фронтенда (Приоритет: P1)

GET `/api/schedule` отдаёт расписание в формате `ISchedule`. Фронтенд загружает расписание из API вместо захардкоженной константы.

**Почему этот приоритет:** Без этого фронтенд не сможет использовать автоматически обновляемое расписание.

**Сценарии приёмки:**

1. **Дано** расписание есть в БД, **Когда** GET `/api/schedule`, **Тогда** ответ 200 с JSON в формате `ISchedule`
2. **Дано** расписание отсутствует в БД, **Когда** GET `/api/schedule`, **Тогда** ответ 200 с пустым расписанием или fallback на последнее известное

---

### US-5 — Фронтенд: загрузка расписания из API (Приоритет: P1)

Фронтенд при загрузке запрашивает `/api/schedule` и кладёт результат в Redux вместо захардкоженного `SCHEDULE`.

**Почему этот приоритет:** Финальный шаг — связать фронт с бэкендом.

**Как проверить независимо:** Открыть сайт → в DevTools Network видно запрос к `/api/schedule` → расписание отображается корректно.

**Сценарии приёмки:**

1. **Дано** бэкенд доступен, **Когда** пользователь открывает сайт, **Тогда** расписание загружается из API
2. **Дано** бэкенд недоступен, **Когда** пользователь открывает сайт, **Тогда** используется кешированное расписание (service worker / localStorage) или захардкоженный fallback

---

### US-6 — Ченжлог изменений расписания на сайте (Приоритет: P2)

Когда парсер обнаруживает изменение расписания — сохранить diff (что было → что стало). На сайте показать ченжлог: дата обновления + список изменений. Пользователь заходит и сразу видит что данные свежие и что конкретно поменялось, без необходимости следить за каналом.

**Почему этот приоритет:** Расписание работает и без ченжлога. Но ченжлог снимает вопросы «а расписание актуальное?» и убирает необходимость вручную писать в канал про каждое мелкое обновление.

**Как проверить независимо:** Парсер нашёл новый файл с изменённым временем → на сайте появилась запись в ченжлоге с конкретными изменениями.

**Сценарии приёмки:**

1. **Дано** парсер обнаружил изменение расписания, **Когда** обновление сохранено, **Тогда** в БД создаётся запись ченжлога с diff (какие рейсы изменились, добавились, удалились)
2. **Дано** есть записи в ченжлоге, **Когда** пользователь открывает сайт, **Тогда** где-то на странице видно «Обновлено: [дата]» и можно посмотреть что изменилось
3. **Дано** расписание не менялось, **Когда** cron отработал, **Тогда** новая запись в ченжлоге не создаётся

---

## Ключевые сущности

- **ScheduleParser**: сервис, который скачивает Word-файл, извлекает таблицы, маппит на `ISchedule`
- **schedule (таблица БД)**: хранит распаршенное расписание как JSON + метаданные (дата обновления, хеш файла-источника)
- **ISchedule**: существующий интерфейс фронтенда — формат данных должен совпадать
- **schedule_changelog (таблица БД)**: записи изменений — дата, описание diff (какие рейсы изменились/добавились/удалились)

## Edge Cases

- Что если перевозчик сменил формат файла или URL? → AI-fallback (GPT API) + алерт в мониторинге
- Что если Word-файл битый или пустой? → Не перезаписывать текущее расписание, залогировать ошибку
- Что если время в файле в 24ч формате, а где-то в 12ч? → Нормализовать при парсинге
- Что если появилась новая остановка, которой нет на фронте? → Добавить в данные, на фронте подхватится через `stopsOptions`
- [УТОЧНИТЬ] Какой точно URL файла на пассажир-онлайн? Нужно исследовать сайт
- [УТОЧНИТЬ] Совпадают ли названия остановок в Word-файле с текущими ключами в `SCHEDULE`? Может потребоваться маппинг

## Критерии приёмки

- [ ] Word-файл скачивается с сайта перевозчика
- [ ] Парсер извлекает расписание в формат `ISchedule`
- [ ] Cron проверяет обновления раз в день
- [ ] Ручной триггер POST `/api/schedule/refresh` работает
- [ ] GET `/api/schedule` отдаёт расписание
- [ ] Фронтенд загружает расписание из API вместо константы
- [ ] При ошибке парсинга текущее расписание не перезаписывается
- [ ] [P2] При обновлении расписания создаётся запись в ченжлоге с diff
- [ ] [P2] На сайте видно дату последнего обновления и список изменений
